# Skill From Masters 精读笔记：三位一体的技能生成系统

> 学习时间：2026-01-25
> 来源：https://github.com/GBSOSS/skill-from-masters
> 核心价值：站在巨人肩膀上创造 Skill——三条路径覆盖所有「巨人」类型

---

## 根节点命题

**创建 Skill 的难点不在格式，而在「知道怎么做才对」——三条路径覆盖所有「巨人」类型：市场现成的、GitHub 开源的、专家方法论的。**

---

## 问题本质

```
创建 Skill 的痛点
    │
    ├─→ 从零开始：不知道最佳实践，踩完所有坑
    │
    ├─→ 信息过载：资源分散在市场、GitHub、书籍...无从下手
    │
    └─→ 质量参差：自己做的 Skill 是「实习生」水平，不是「专家」水平

根本问题：信息差 = 你不知道「巨人」在哪
```

**解法**：

```
三条路径，覆盖所有「巨人」
    │
    ├─→ 路径1：search-skill → 巨人类型：已做好放市场的 Skill
    │
    ├─→ 路径2：skill-from-github → 巨人类型：GitHub 上的优质开源项目
    │
    └─→ 路径3：skill-from-masters → 巨人类型：领域专家的方法论
```

## 核心机制精解

### 机制 1：三条路径的优先级

**成本递增原则**：
1. **市场现成的**：零成本，直接用
2. **GitHub 学习的**：中等成本，需要理解源码
3. **专家方法论创造的**：高成本，需要深度研究

> 「先看有没有轮子，再考虑造轮子」

### 机制 2：5 层收窄框架

非技术类 Skill 必须「足够窄」，否则方法论太泛、输出平庸。

| 层级 | 问什么 | 示例 |
|------|--------|------|
| Layer 1 | 哪个领域？ | 产品决策 / 招聘决策？ |
| Layer 2 | 5W1H 约束？ | 谁用？什么输出？ |
| Layer 3 | 具体场景？ | Sprint 规划 / 季度路线图？ |
| Layer 4 | 不包含什么？ | ❌ 危机响应 ❌ 高管汇报 |
| Layer 5 | 真实案例？ | 上月 API 限流 PRD... |

**停止收窄的 5 个条件**：
1. ✅ 专家会有独特建议？
2. ✅ 能判断输出「优秀」还是「平庸」？
3. ✅ 有特定约束和失败模式？
4. ✅ 用户描述了真实场景？
5. ✅ 明确排除了相关任务？

### 机制 3：三层方法论搜索

```
Layer 1: 本地数据库 → 查 methodology-database.md
Layer 2: Web 搜索专家 → "[domain] framework methodology"
Layer 3: 深挖一手资源 → "[expert] transcript"
Layer 4: 持续迭代 → 不确定就继续搜
```

### 机制 4：Skill 类型分类（11 种）

| 用户说... | 可能类型 |
|-----------|----------|
| 帮我写... | Generation |
| 帮我理解... | Insight / Summary |
| 帮我决定... | Decision |
| 帮我评估... | Evaluation |
| 帮我找出为什么... | Diagnosis |
| 帮我说服... | Persuasion |
| 帮我计划... | Planning |

### 机制 5：黄金案例 + 反模式

- **黄金案例**：定义「什么叫做得好」
- **反模式**：编码「千万别这么干」

PRD 示例：
- 黄金案例：Intercom PRD 模板、Linear 产品规格
- 反模式：没定义问题就跳方案、缺少成功指标

### 机制 6：「学知识」vs「包装工具」

| 包装工具 | 学知识 |
|----------|--------|
| 调用命令行工具 | 理解核心算法 |
| 依赖环境安装 | 随处可用 |
| 工具升级可能挂 | 核心原理不变 |

## 可迁移的提效经验

| 原做法 | 优化后 |
|--------|--------|
| 直接写 Skill | 先 search-skill 看有没有现成的 |
| 凭感觉定义任务 | 用 5 层收窄框架系统定义 |
| 不分类 | 先识别 Skill 类型，再选方法论 |
| 只搜一次 | 三层搜索 + 迭代到「非常清晰」 |
| 写完就用 | 设计测试场景 → 迭代优化 |

### 立即可用的技巧

1. **先市场后创造**：search-skill 找现成的
2. **任务收窄**：非技术 Skill 定义到「专家会有独特建议」的粒度
3. **模型分级**：非技术用 Opus，技术用 Sonnet
4. **黄金案例**：先找「做得好的例子」定义质量标准
5. **反模式**：编码「千万别这么干」

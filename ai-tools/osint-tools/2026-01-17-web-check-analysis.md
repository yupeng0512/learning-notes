## 🎯 这个项目解决什么问题？

### 痛点场景

1. **安全研究员**：做渗透测试前需要收集目标信息，要开十几个工具分别查询
2. **网站开发者**：想了解竞品网站用了什么技术栈，需要手动分析
3. **运维人员**：排查网站问题时，需要逐一检查 DNS、SSL、Headers 等配置
4. **SEO 从业者**：分析网站技术健康度，要用多个工具拼凑数据

### Web-Check 的解决方案

**一个入口，全维度分析**：输入 URL → 自动执行 30+ 种检查 → 结构化展示结果

```
传统方式：
nslookup → whois → curl -I → nmap → traceroute → ...（10+ 工具）

Web-Check：
输入 URL → 点击分析 → 一页看完所有结果
```

### 架构设计亮点

| 设计决策 | 理由 | 学习点 |
|----------|------|--------|
| **Astro 作为主框架** | 静态生成 + 按需 hydration，首屏快 | 内容为主的工具站首选 Astro |
| **多框架混用** | React + Svelte 共存 | Astro 的 Islands 架构支持 |
| **前后端分离但同仓库** | 简化部署，一个镜像搞定 | Monorepo 思维 |
| **Puppeteer 无头浏览器** | 截图/执行 JS/绕过反爬 | OSINT 工具必备 |
| **Express-rate-limit** | 防止 API 滥用 | 公开服务必加限流 |

### 重点检查项详解

#### 1. IP 信息 → 基础中的基础

```bash
# 等效命令
nslookup example.com
dig example.com A

# Web-Check 额外提供
- IP 地理位置
- ASN 信息
- 托管商识别
- 同 IP 其他域名
```

**学习价值**：IP 是所有进一步调查的起点

#### 2. SSL 证书链 → 安全第一道防线

```
检查内容：
├── 证书颁发机构 (CA)
├── 证书有效期
├── 证书链完整性
├── 证书透明度日志
└── 组织信息（有时包含公司名）
```

**学习价值**：SSL 证书常泄露关联子域名、公司实体

#### 3. 技术栈识别 → Wappalyzer 集成

```json
// 示例输出
{
  "CMS": "WordPress 6.4",
  "JavaScript 框架": "React 18",
  "Web 服务器": "Nginx",
  "CDN": "Cloudflare",
  "分析工具": "Google Analytics"
}
```

**学习价值**：知道对方用什么技术，就知道可能有什么漏洞

#### 4. HTTP 安全头 → 快速评估防护水平

| 安全头 | 作用 | 缺失风险 |
|--------|------|----------|
| `Strict-Transport-Security` | 强制 HTTPS | 中间人攻击 |
| `Content-Security-Policy` | XSS 防护 | 脚本注入 |
| `X-Frame-Options` | 防点击劫持 | 钓鱼攻击 |
| `X-Content-Type-Options` | MIME 嗅探防护 | 类型混淆攻击 |

**学习价值**：安全头是最低成本的防护，但很多站点都没配置

#### 5. 开放端口 → 攻击面识别

```
常见端口及含义：
21   - FTP（文件传输，常有漏洞）
22   - SSH（远程管理）
80   - HTTP
443  - HTTPS
3306 - MySQL（不应公开）
8080 - 备用 HTTP / 管理后台
```

**学习价值**：非必要端口不应暴露在公网

## 🛠️ 实战：本地部署体验

### 方式一：Docker（推荐）

```bash
# 最简单的方式
docker run -p 3000:3000 lissy93/web-check

# 带环境变量（启用更多功能）
docker run -p 3000:3000 \
  -e GOOGLE_CLOUD_API_KEY=xxx \
  -e REACT_APP_SHODAN_API_KEY=xxx \
  lissy93/web-check
```

### 方式二：源码运行

```bash
git clone https://github.com/Lissy93/web-check.git
cd web-check
yarn install
yarn dev  # 开发模式，前后端并行启动
```

### 可选 API Keys（解锁更多功能）

| 环境变量 | 功能 | 获取方式 |
|----------|------|----------|
| `GOOGLE_CLOUD_API_KEY` | Lighthouse 质量指标 | Google Cloud Console |
| `REACT_APP_SHODAN_API_KEY` | 关联主机名 | shodan.io |
| `SECURITY_TRAILS_API_KEY` | IP 关联组织 | securitytrails.com |
| `URL_SCAN_API_KEY` | 站点杂项信息 | urlscan.io |

## ⚠️ 常见误区

| 误区 | 正确做法 |
|------|----------|
| ❌ OSINT 工具只能用于攻击 | ✅ 同样适用于防守（了解自己的暴露面） |
| ❌ 检查结果越多越好 | ✅ 关键是理解每个检查项的安全含义 |
| ❌ 公开部署不加限流 | ✅ 必须配置 `API_ENABLE_RATE_LIMIT` |
| ❌ 用 Web-Check 扫描他人网站是合法的 | ✅ 仅用于自己的资产或获得授权的目标 |
| ❌ 所有 API Key 都是必须的 | ✅ 核心功能无需 Key，Key 只是增强 |

## 📋 一页纸总结

### 核心要点（5 条）
1. **一站式 OSINT**：30+ 检查项覆盖侦察全流程，一个入口替代十几个工具
2. **Astro + Express 架构**：前端静态生成 + 后端 API，部署灵活
3. **Puppeteer 核心**：无头浏览器实现截图、JS 渲染、反爬绕过
4. **安全头是低垂的果实**：很多网站连基本的 HSTS、CSP 都没配置
5. **合法合规使用**：仅用于自己的资产或获得授权的目标

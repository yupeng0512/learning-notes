## 概述

Boris Cherny 是 Claude Code 的创造者。他的配置"极其朴素"，因为 Claude Code 本身就能完美开箱即用。

**核心理念**：
> 使用 Claude Code 并没有所谓的唯一正解。我们在构建它时就特意设计成这样，你可以按照自己喜欢的方式去使用、定制，甚至大肆改造它。

## 13 个技巧详解

### 技巧 1：并行运行多个实例

**做法**：
- 终端里并行运行 **5 个 Claude 实例**
- 标签页按 1-5 编号
- 利用**系统通知**提醒何时需要人工输入

**核心思想**：并行 > 串行，最大化利用等待时间

```
┌─────────────────────────────────────────────┐
│  Tab 1    Tab 2    Tab 3    Tab 4    Tab 5  │
│  Claude   Claude   Claude   Claude   Claude │
│  任务A    任务B    任务C    任务D    任务E   │
└─────────────────────────────────────────────┘
           ↓ 系统通知 ↓
    "Tab 3 需要你的输入"
```

---

### 技巧 2：终端 + 网页端混合协作

**做法**：
- 网页端同时运行 **5-10 个 Claude 会话**
- 用 `&` 将本地会话推送到网页端
- 用 `--teleport` 在终端和网页间切换
- 手机端（Claude iOS）开启会话，稍后查看进度

**场景示例**：
```bash
# 推送到网页端继续
claude & 

# 在终端和网页间切换
claude --teleport
```

**核心思想**：多端协作，随时随地推进任务

---

### 技巧 3：始终使用 Opus 4.5 + Thinking

**选择**：带"思考过程（thinking）"的 **Opus 4.5**

**为什么不用更快的 Sonnet？**

| 对比项 | Sonnet | Opus 4.5 |
|--------|--------|----------|
| 速度 | 更快 | 更慢 |
| 体积 | 更小 | 更大 |
| 听劝程度 | 需要更多引导 | 更听劝 |
| 工具使用 | 一般 | 更擅长 |
| **最终效率** | 较低 | **更高** |

**核心洞见**：
> 虽然它比 Sonnet 更大、更慢，但因为它更听劝（需要引导的地方少）且更擅长使用工具，从结果来看，它几乎总是比用小模型效率更高。

### 技巧 4：团队共享 CLAUDE.md

**做法**：
- 团队共用一个 `CLAUDE.md` 文件
- 提交到 git 仓库，全员共享
- **每周多次更新**
- 发现 Claude 做错的事，就记录进去

**本质**：团队级别的"错题本"

```markdown
# CLAUDE.md 示例

## 项目约定
- 使用 TypeScript strict 模式
- 测试文件放在 __tests__ 目录

## 已知问题修正
- ❌ 不要在 useEffect 里直接调用 async 函数
- ✅ 应该定义内部 async 函数再调用

## 代码风格
- 组件使用 PascalCase
- hooks 使用 use 前缀
```

### 技巧 5：PR 中 @.claude 更新 CLAUDE.md

**做法**：
- 代码审查时在同事 PR 上 `@.claude`
- 要求将某些内容添加到 `CLAUDE.md`
- 使用 Claude Code 的 **GitHub Action**

**安装方式**：
```bash
/install-github-action
```

**核心思想**：让 CLAUDE.md 的更新成为代码审查的一部分

### 技巧 6：计划模式 → 自动接受模式

**工作流**：

```
1. 双击 Shift+Tab 进入"计划模式"
      ↓
2. 与 Claude 来回沟通，打磨方案
      ↓
3. 满意后，切换到"自动接受修改模式"
      ↓
4. Claude 一次性完成实现
```

**核心洞见**：
> 一份好的计划至关重要！

**类比**：先画蓝图，再施工

---

### 技巧 7：斜杠命令自动化内循环

**做法**：
- 每天重复多次的工作流 → 斜杠命令
- 命令存放在 `.claude/commands/` 目录
- 提交到 git，全队共享

**示例：/commit-push-pr**

```bash
# .claude/commands/commit-push-pr.md
# 每天使用几十次

# 预先计算 git 状态（内联 bash）
# 避免与模型不必要的来回确认
```

**优势**：
- 省去重复输入 Prompt
- Claude 自己也能调用这些命令
- 预计算信息，运行飞快

---

### 技巧 8：子智能体（Subagents）

**常用子智能体**：

| 子智能体 | 用途 |
|----------|------|
| `code-simplifier` | 完成后简化代码 |
| `verify-app` | 端到端测试的详细指令 |

**定位**：将 PR 中最常见的流程自动化

**与斜杠命令的区别**：
- 斜杠命令：简单的预设 Prompt
- 子智能体：复杂的多步骤流程

---

### 技巧 9：PostToolUse 钩子格式化代码

**做法**：使用 `PostToolUse` 钩子自动格式化

**为什么需要？**
- Claude 生成的代码格式已经很不错
- 但钩子能搞定**最后 10% 的细节**
- 避免之后在 CI 中报错

**配置示例**：
```json
// .claude/hooks/post-tool-use.json
{
  "onFileWrite": "prettier --write {file}"
}
```

### 技巧 10：权限管理策略

**不推荐**：`--dangerously-skip-permissions`

**推荐做法**：
1. 使用 `/permissions` 预先批准安全命令
2. 配置保存在 `.claude/settings.json`
3. 全队共享

```json
// .claude/settings.json
{
  "allowedCommands": [
    "npm test",
    "npm run build",
    "git status",
    "git diff"
  ]
}
```

**核心思想**：安全 + 高效的平衡

### 技巧 11：MCP 集成各种工具

**Claude Code 帮 Boris 操作的工具**：

| 工具 | 用途 |
|------|------|
| Slack MCP | 搜索或发帖 |
| BigQuery (bq) | 运行查询回答分析问题 |
| Sentry | 抓取错误日志 |

**配置方式**：
- 定义在 `.mcp.json`
- 全队共享

```json
// .mcp.json
{
  "mcpServers": {
    "slack": {
      "command": "npx",
      "args": ["@anthropic/mcp-server-slack"]
    }
  }
}
```

### 技巧 12：超长任务的三种方案

| 方案 | 做法 | 适用场景 |
|------|------|----------|
| A | 提示 Claude 完成后用**后台智能体验证** | 一般长任务 |
| B | 使用 **Stop 钩子**确定性执行验证 | 需要可靠验证 |
| C | 使用 **ralph-wiggum 插件** | 特殊场景 |

**配合使用**：
- `--permission-mode=dontAsk`
- 或在沙箱里开启危险跳过模式

**效果**：Claude 不被打断，自主完成

### 技巧 13：给 Claude 验证方法（最重要！）

> **这可能是让 Claude Code 产出高质量结果最重要的事**

**核心原则**：
> 只要有反馈闭环，最终结果的质量会提升 **2-3 倍**

**Boris 的做法**：
- Claude 使用 **Chrome 扩展程序**测试每个改动
- 打开浏览器，测试 UI
- 不断迭代直到代码运行正常且交互体验良好

**不同领域的验证方式**：

| 领域 | 验证方式 |
|------|----------|
| 后端 | 运行 bash 命令 |
| 测试 | 运行测试集 |
| 前端 | 浏览器测试 UI |
| 移动端 | 模拟器测试 |

**核心洞见**：
> 务必投入精力把这个验证环节做扎实。

## 核心洞见

### 1. 简单即有效

> 我的配置可能会让你大吃一惊，因为它极其朴素。

创始人的配置反而最简单，说明 Claude Code 的默认设计已经足够好。

### 2. 反馈闭环是关键

> 只要有反馈闭环，最终结果的质量会提升 2-3 倍。

这是整篇文章最重要的一条。没有验证 = 盲飞。

### 3. 团队共享配置

多个配置文件都强调"全队共享"：
- `CLAUDE.md`
- `.claude/commands/`
- `.claude/settings.json`
- `.mcp.json`

### 4. 大模型 > 小模型（在复杂任务上）

> 虽然 Opus 更慢，但因为更听劝、更擅长工具使用，最终效率更高。

不要被"快"迷惑，要看最终结果。
